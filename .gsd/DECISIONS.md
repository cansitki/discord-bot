| D007 | M001/S04 | pattern | Server design tool approach for Claude | Single `propose_server_design` tool that returns full plan JSON, not individual create_channel/create_role tools | Users must approve the entire plan before any Discord resources are created. Individual CRUD tools risk Claude creating channels before approval. A single propose tool lets Claude reason about the whole structure, produces one approvable artifact, and keeps the execution phase bot-controlled (no Claude involvement in the build step). | Yes — could add individual tools later for incremental modifications (e.g., "add one more channel") |
| D008 | M001/S04 | data | Storage for pending server design proposals | In-memory dict keyed by (guild_id, message_id), not a database table | Proposals are short-lived (user clicks Approve/Cancel within minutes). Persisting across restarts adds complexity for minimal value — if the bot restarts, users can re-request. The verification cog's DynamicItem buttons already handle the "button survives restart" case, but the proposal data itself doesn't need persistence. | Yes — could move to SQLite if proposal abandonment becomes a problem |
| D009 | M001/S05 | pattern | How cogs expose tools to Claude via ai.py | Duck-typed protocol: any cog with `get_tools() -> list[dict]` and `handle_tool_call(name, input, message) -> str` methods gets its tools automatically included in Claude API calls | The hardcoded `ServerDesignCog` check in ai.py doesn't scale to multiple tool-providing cogs. A duck-typed protocol (hasattr check) avoids import coupling between ai.py and each tool cog, allows future cogs to add tools by just implementing two methods, and keeps ai.py as a generic router rather than a registry of specific cogs. No ABC needed — Python duck typing is sufficient for 2-3 cogs. | Yes — could formalize with a Protocol/ABC if the cog count grows beyond 5 |
| D010 | M001/S01/T01 | pattern | Config dataclass mutability | Frozen dataclass for Config to prevent accidental mutation of credentials at runtime | Config holds bot token and API key. A frozen dataclass prevents any code from accidentally modifying these values after load. The small inconvenience of immutability is worth the safety for credential-bearing objects. | Yes |
| D011 | M001/S02/T01 | pattern | Persistent button pattern for verification cog | DynamicItem subclasses (ApproveButton, DenyButton) with member_id encoded in custom_id pattern, registered via bot.add_dynamic_items() | Verification buttons encode the target member_id in the custom_id (e.g. verify:approve:12345). add_view() requires static custom_ids — it can't handle dynamic per-member IDs. DynamicItem (discord.py v2.4+) uses regex templates to match and reconstruct state from custom_id, making it the correct API for buttons that survive restarts and carry per-interaction state. | No — DynamicItem is the correct discord.py API for this use case |
| D012 | M001/S04 | pattern | Partial failure handling in server design build executor | Report created and failed resources without rollback — do not delete successfully created resources on partial failure | Rolling back partially-created Discord resources (deleting channels/roles on error) adds complexity and risks leaving the server in a worse state if the rollback itself fails. Since this is a small friend group bot, manual cleanup of a few extra channels is trivial. The build summary message explicitly lists what was created and what failed, giving the admin enough information to act. | Yes — could add rollback if server design is used frequently and partial failures cause confusion |
| D013 | M001/S06 | operability | Railway restart policy and healthcheck configuration | ON_FAILURE restart with 10 max retries, no healthcheck endpoint | The bot is a Discord Gateway worker, not an HTTP server — healthcheck paths don't apply. ON_FAILURE with 10 retries means the bot auto-restarts on crashes but stops on clean shutdown (e.g., SIGTERM from Railway during redeploy). This avoids infinite restart loops on fatal configuration errors while recovering from transient failures. | Yes |
| D014 | M001/S06 | operability | Pre-deployment verification gate pattern | verify-deploy.sh with 18 checks across config, imports, deps, and migrations | A bash script that imports all bot modules, validates config files, and checks migrations catches deployment-breaking issues (missing dependencies, import errors, invalid config) before pushing to Railway. Runs in <1s, doesn't need real tokens, and serves as the canonical pre-deploy command. Any new cog or module must be added to the script's import list. | Yes |
| D015 | M002 planning | arch | GitHub App authentication dependencies | PyJWT with cryptography extra (PyJWT[crypto]) for RS256 JWT generation | GitHub App authentication requires generating RS256-signed JWTs from the App's private key. PyJWT[crypto] is the standard Python library for this. The cryptography extra provides the RSA backend needed for RS256 signing. This is a new dependency (aiohttp and PyJWT[crypto]) but both are well-maintained and lightweight. | Yes |
| D016 | M002 planning | library | GitHub API client library choice | Raw httpx with thin GitHubClient wrapper, not PyGithub or gidgethub | httpx is already a dependency (used by assistant cog's fetch_url). PyGithub is synchronous and would block the event loop. gidgethub adds unnecessary abstraction for the 5-6 GitHub API endpoints we need (repos, issues, pulls, commits). A thin async wrapper around httpx keeps the dependency count low and gives full control over JWT auth and installation token management. | Yes — could switch to gidgethub if the GitHub API surface grows significantly |
| D017 | M002 planning | arch | Webhook receiver architecture | aiohttp web server embedded in the bot process via AppRunner/TCPSite in setup_hook, not a separate Railway service | Embedding the webhook receiver in the bot process is simpler than running a separate service: single deployment, shared database connection, direct access to bot.get_channel() for event routing. The aiohttp AppRunner/TCPSite pattern is designed for embedding in existing async applications. The downside (bot crash kills webhook delivery) is acceptable for a friend group bot. Railway provides PORT env var for the web process. | Yes — could split into separate Railway service if reliability becomes critical |
| D018 | M002 planning | scope | R008 (PR summaries + Code Q&A) scope for M002 | Deferred out of M002 to a later milestone | The milestone context document explicitly excludes R008 from M002 scope. PR summaries and code Q&A require significant context-window management and GitHub file-tree traversal that would add complexity without delivering the core user request (issue creation + webhooks). Better to ship a solid core GitHub integration first and layer on intelligence features later. | Yes — planned for a future milestone |
| D019 | M002/S01/T02 | pattern | GitHub API test mocking approach | httpx.MockTransport injected via http_client constructor parameter, not pytest-httpx or responses library | httpx.MockTransport is built into httpx (already a dependency), provides clean async mocking without extra test dependencies, and dependency injection via http_client parameter keeps the production GitHubClient constructor simple while making tests fully controllable. All GitHub API test patterns (JWT exchange, get_repo, future create_issue/list_pulls) reuse this approach. | Yes |
